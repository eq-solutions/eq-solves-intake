EQ Intake — Architecture Overview

The layer that stops
people retyping the same thing twice

Trade subbies spend hours every week moving data between systems that don't talk to each other. EQ Intake is the plumbing in the middle — data goes in once, comes out in whatever format every other system needs.

😤

The real problem EQ solves

An apprentice writes their details on paper at 6:30am. The office types it into SimPRO. Someone else copies it into the Equinix portal. The bookkeeper puts hours into Xero. The insurance renewal needs all of it again. That's the same data, typed 4–5 times, by 4–5 people, into 4–5 systems. Every retype is a chance to get it wrong. EQ removes every retype except the first one.

Under the Hood
What makes it reliable
You don't need to know this to use EQ. But here's why the architecture is built this way.
🔒
Row-level security
Each row in the database carries a tenant_id. The database itself refuses cross-tenant reads — not just the app code. SKS data is physically isolated.
↩️
Atomic rollback
Every import is tagged with an intake_id. One RPC call undoes the entire batch cleanly — no partial states, no manual SQL cleanup tickets.
🏷️
Schema versioning
Every row is tagged with the schema version that produced it. Breaking changes get a major version bump and a migration window — old data is never corrupted silently.
💸
AI cost caching
Before calling AI to map columns, EQ hashes the spreadsheet shape. Same shape as last time? Skip the AI call entirely. Routine imports cost nothing after the first run.
⚡
Fast validation
10,000 rows × 50 fields validate in under 2 seconds. Same engine runs inside Cards on the phone, inside Format on the desktop, inside every intake path.
🚩
No silent drops
Every input row gets an output — either committed, flagged, or rejected with a reason. EQ cannot silently lose your data. Every drop is visible to the operator.

What we explicitly don't build (and why)

Another job-management system — SimPRO does that. We feed it, we don't replace it.
Another accounting platform — Xero and MYOB do that fine.
A new compliance management system — too broad, too far from the pain.
Anything that asks users to switch off what they're already using.
Features that nobody has a named moment of pain for. No moment, no build.
Paywalls on safety features — inductions, SWMS, prestarts. People die when this goes wrong.
