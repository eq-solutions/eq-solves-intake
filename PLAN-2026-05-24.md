# Next-steps plan — 2026-05-24

## The mission, restated

EQ is being built for Royce's operations at SKS (NSW branch and beyond). Not to sell to
random subbies. Not to validate against beta testers. The architecture is the product —
built correctly once, not incrementally justified by customer count.

**The measure of success:** does this make Royce's actual operations at SKS NSW
meaningfully better? The Equinix audit that takes 3 hours → 20 minutes. The bookkeeper
who retypes timesheets on Friday → gets them clean by Thursday. The apprentice doing
the same induction at every data centre → does it once, exports it everywhere.

If it scales beyond SKS eventually, it does so because it's genuinely better than
what exists — not because anyone went looking for beta testers.

---

## What's blocking everything else — fix these first

Three live bugs that violate the core brief ("every row in deserves a row out").
These land in week 1. No other new work before they're closed.

**C1** — [`validate.ts:185`](eq-platform/packages/eq-validation/src/validate.ts) silently
drops all rows past 100k. No rejection record emitted. A 110k-row file loses 10k rows
with no trace. Fix: emit `(rejected, reason: cap_exceeded)` for every overflow row,
or paginate. Don't truncate.

**C2** — [`commit-canonical.ts:337`](eq-platform/packages/eq-intake-demo/src/canonical/commit-canonical.ts)
lets FK-missing rows land in `flagged` or `valid` instead of `rejected`. This is the
mechanism that lost 72 Equinix sites in May. Fix: FK miss must always produce
`(rejected, reason: fk_no_match)`. No exceptions.

**C3** — [`rollup.mjs:364`](demos/simpro-customer-rollup/rollup.mjs) logs orphan contacts
to console only, never emits them to output. Fix: mirror the orphan-site pattern,
emit with `[orphan]` marker. Same fix in [`template.ts`](eq-platform/packages/eq-intake-demo/src/rollup/template.ts)
for dual-engine parity.

## Three risks most likely to derail this

### R1. Canonical Supabase provisioning gets stuck on billing approval

The old plan deferred this. The new plan needs it in week 1. If billing approval takes
longer than expected, weeks 3–10 all slip. Unblock early: Royce confirms billing
approval at the start of week 1, not after the bugs are fixed.

### R2. The 72-site silent-drop pattern reappears in the Equinix profile

Any PR that adds a new `continue` / `filter()` / silent skip in a row-processing
loop without an accompanying rejection-record emit reintroduces the pattern. Watch
for this in every intake, rollup, and format-ui derive change.

**Signal:** any diff during weeks 3–4 that modifies `derive/` without a corresponding
test for the `rejected` bucket.

### R3. Cards migration becomes a multi-week scope creep

The bridge doc says "one deliberate weekend of work." If it expands — schema mismatch
surprises, feature gaps between old Cards Supabase and canonical shape, re-building
things that already exist — it will eat weeks 7–10.

**Control:** do a dry-run schema comparison before starting the migration. Know the
exact diff between Cards' tables and canonical schemas before writing a line.

## What's explicitly NOT in this plan

- Selling EQ to external subbies. That's not what this is.
- Waiting for "customer pressure" to justify architecture investment. Build it right.
- A new job-management system, accounting platform, or field scheduling tool.
- Phase 4/5 backlog items that don't connect to a named SKS NSW moment.
- Per-tenant provisioning automation — wait until the second tenant is real, not before.

If something here turns out to be wrong at week 6, re-anchor against `EQ-AS-CONDUIT.md`,
write a one-page revision to this plan, and keep going. The plan is a guide, not a contract.
