# Next-steps plan — 2026-05-22

## The single highest-leverage next thing

**Get EQ Cards onto the phone of one outside-SKS sparkie, watch them use it for a real induction, fix what breaks, repeat with the next one.**

That's the whole plan. Everything else either makes that possible or stays cold.

The named moment of pain: a sparkie shows up to a data centre at 6:30am, opens whatever induction portal the principal contractor demands, types the same details he typed yesterday at a different site for a different client, takes a phone photo of his licence with shaking hands because his coffee hasn't kicked in. He's done it 30+ times this year. He'll do it again tomorrow.

Cards exists to make that moment 30 seconds instead of 8 minutes. The OCR works on Royce's licence. It hasn't been tested on anyone else's. Nobody who isn't Royce or Emma has installed it yet. Every claim about whether Cards is good *or* bad is currently speculation. One real outside-SKS sparkie using it removes that speculation faster than another month of building does.

## The validation milestone

**5 outside-SKS sparkies have each used Cards at least twice, on their own initiative, with no Royce-prompt in between.**

The milestone is measurable and externally observable: there's an event log entry for each install + event log entries for each induction + a known list of which installs are from people who aren't Royce, Emma, or current SKS staff. The "on their own initiative" half is harder — if Royce had to text-prompt each second use, the install didn't actually stick. PostHog second-use events without a precipitating "hey try Cards again" Slack/SMS is the signal.

Anything less than this gate — "tests are passing," "Cards is built," "the OCR works," "demos run cleanly" — is plumbing, not validation.

## Pushback on Royce's "Equinix → SimPRO is the obvious first deep integration"

Royce's instinct: one door out, deep, before any universal anything. Equinix audit format → SimPRO export, because SKS lives in both. That instinct is **right about depth** and **right about avoiding universal engines** — the audit confirmed Format-the-package is already healthy enough that adding the Equinix profile wouldn't trigger the N×M creep.

But Equinix→SimPRO is SKS-internal. It doesn't move the 5-outside-SKS-sparkies gate. It validates the conduit thesis on a workflow that already works (just slowly). Cards validates the conduit thesis on the part that doesn't work at all yet — first-touch from someone who isn't Royce.

**Recommendation: Equinix→SimPRO runs in parallel as a secondary workstream once Cards has stabilised on 1 outside user.** It's the workstream Royce can advance with his own pain as validator (`feedback_royce_is_the_user`) during weeks where Cards distribution is stalled. It doesn't gate anything. The primary gate is still the 5 sparkies.

If Royce disagrees and wants Equinix→SimPRO primary, the cost is: weeks 1-12 produce a deep SKS-internal integration but zero outside-SKS validation. EQ stays a thing Royce built for SKS. The 5-sparkie gate slides past 90 days.

## Three risks most likely to derail this

### R1. Cards doesn't reach outside-SKS users — distribution is the hardest part

The whole plan assumes Royce can put Cards on 5 strangers' phones inside 90 days. He doesn't have a distribution channel. The likely path is word-of-mouth from the first sparkie; if the first sparkie doesn't tell anyone, the second sparkie doesn't materialise.

**Early signal:** end of week 4 and the only Cards installs from outside SKS are sparkie #1. No organic referrals. If that's the state, the assumption that "Cards is valuable enough to share" is the thing in question, not the plan's pacing.

### R2. The 72-site silent-drop pattern bites a real outside user before the fix lands

The fix is week 1, the first user is week 2. If a new intake code path lands during weeks 3–10 without the `(rejected, reason)` discipline enforced — particularly if Equinix→SimPRO introduces a new reshape-out direction — the silent-drop pattern can reappear.

**Early signal:** any PR (here or in eq-solves-service) that adds a new `continue` / `filter()` / silent-skip in a row-processing loop without an accompanying rejection-record emit. Watch for this in code review on every PR that touches intake, format-ui derive, or commit-canonical.

### R3. Equinix→SimPRO turns into universal-Format-engine work

The audit found Format-the-package is healthy — 3 narrow profiles, no plugin system. Adding the Equinix profile in week 5 means a 4th profile. The temptation will be to factor out a shared pattern, generalise the input-shape discriminator, introduce a profile-config schema. That's exactly the universal-engine drift the conduit thesis rules out.

**Early signal:** any commit during weeks 5–10 that modifies `eq-platform/packages/eq-format-ui/src/derive/types.ts` or `registry.ts` to make the input-shape system more polymorphic. The Equinix profile should be a single file that imports `DeriveProfile`, exports a function, gets added to the registry's hardcoded Map. If the diff is more than that, it's drift.
