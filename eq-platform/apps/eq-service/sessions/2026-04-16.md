## Item 11 — Data-quality audit in CI

### New

- `.github/workflows/data-quality.yml` — new GitHub Action. Runs on PRs touching `audits/**` or `supabase/migrations/**`, on pushes to main, daily at 08:30 UTC, and on manual dispatch. POSTs `audits/run.sql` as `{query: ...}` to `https://api.supabase.com/v1/projects/{ref}/database/query` using `SUPABASE_ACCESS_TOKEN` + `SUPABASE_PROJECT_REF` secrets (same secrets the existing `supabase-advisors.yml` workflow uses — **no new secrets required**). Parses the result with jq, tabulates ERROR/WARN counts, writes a full summary to `$GITHUB_STEP_SUMMARY`, uploads `audit.json` as an artifact, and fails the job **only** when any `level = 'ERROR'` row has `fail_count > 0`.

### Modified

- `audits/CHECKS.md` — appended a "CI behaviour" section documenting:
  - what the workflow runs on
  - the ERROR-fails-build / WARN-tabulates-only rule
  - how to waive a check (fix data, demote to WARN with a comment, or tighten the check to exclude a known exception — no ad-hoc workflow skips)
  - how to re-baseline after an intentional migration that changes expected counts

### Not touched

- `audits/run.sql` itself — the existing output shape (`check_id | level | result | fail_count | detail`) already fits the parser. No SQL changes.

## Item 1 — NSX/MCCB Steps 2 & 3 to ACB parity

### New server actions

Appended to `app/(app)/nsx-testing/actions.ts`:

- `saveNsxVisualCheckAction(testId, items)` — mirrors `saveAcbVisualCheckAction`.
