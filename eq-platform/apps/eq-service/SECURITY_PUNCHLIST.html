3. Before Equinix / external rollout Medium

Not urgent for internal pilot, but required before handing the app to a client.

Netlify branch deploy from a staging branch pointing at a separate Supabase project (or a Supabase branch). Gives you somewhere to test migrations and run authenticated scans without touching prod.
Create a dedicated test user, configure ZAP with a Form-Based Authentication context, run an active scan. This is where real vulns (IDOR, broken access control, SQL injection through params) would surface. Unauthenticated passive scanning misses all of that.
Create two tenants with overlapping data, then run integration tests that log in as Tenant A and try to read/write Tenant B's data via every endpoint. Every attempt should 403 or return empty. This is the single highest-value security test for a multi-tenant app.
Brute-force protection for /auth/signin and password reset. Supabase Auth has some built-in, but add Netlify Edge Functions or middleware limits as defense-in-depth.
Without it you only hear about bugs when users report them, and you miss silent failures entirely — including security-relevant ones like repeated 401s on sensitive endpoints.
One page: how to rotate Supabase keys, how to force-logout all users, how to restore a Postgres backup, who to tell (Equinix / Schneider security contacts). Must exist before you need it.
Supabase takes daily backups but you won't know restore actually works until you try it on a branch. Do a full restore to a Supabase branch once, confirm data looks right, document the steps.
CLAUDE.md says every mutation writes an audit log entry. Spot check: do DELETEs, role changes, and file uploads all land in the audit_log table? If Equinix ever asks "who did X?" you need an answer.
