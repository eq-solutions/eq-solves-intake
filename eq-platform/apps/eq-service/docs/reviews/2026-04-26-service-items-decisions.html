01Dashboard switcher

Top-right "All works" toggle — currently confusing. You wanted clarity on what each side actually shows.

Dashboard
Replace "All works" with: All Active Work ↔ Assigned to Me
Build
Your note
"Needs to be clearer — All sites / Logged in username sites (make it make sense Claude!!)"
Decision
Task-centric toggle. "All Active Work" shows everything open in the tenant; "Assigned to Me" filters to checks & work orders where you're the named assignee.
Why
Site-based filters duplicate what the Sites page already does. Dashboard is about what needs doing — so the filter should follow tasks, not geography.
Build
  • Add assignee_user_id to ACB tests, NSX tests, general tests, work orders (migration)
  • Toggle component in dashboard header — defaults to "Assigned to Me" if user has any assigned items, else "All Active Work"
  • KPI cards, charts, lists all respect the toggle
  • Persist choice per-user in profiles.dashboard_scope

02Loading states

No feedback when you click — feels broken even when it's working.

Overall
Add visible loading state to every action
Fix
Decision
Spinners on buttons during server actions; skeleton loaders on tables/cards during data fetch; subtle top progress bar on route changes.
Why
Perceived performance is half the battle. Without feedback, users double-click, get duplicate writes, and assume the app is dead.
Build
  • Wrap all useTransition server actions with disabled-state + spinner pattern
  • Reusable <LoadingButton> component to standardise this
  • Add app/loading.tsx Suspense boundaries on every route group
  • NProgress-style top bar via Next 16 router events

03Logo upload silent fail

Customer logo says "saved" but doesn't update.

Sites
Fix logo upload — confirm save AND visibly update
Fix
Decision
Reproduce → trace → fix. Likely either (a) revalidatePath() not called on the customer detail route, (b) image URL cached at CDN, or (c) update writes to the wrong column.
Build
  • Verify the upload action returns the new public URL and updates customers.logo_url
  • Force revalidatePath('/customers/[id]') + revalidatePath('/sites/[id]')
  • Cache-bust by appending ?v=<updated_at> to the image src
  • Toast: "Logo updated" only fires on confirmed success, not on submit

04Contacts

No clear way to add contacts today.

Contacts
Per-site & per-customer with optional roles
Build
Decision
Contact rows attach to a customer or a specific site. Optional role tag (Site Contact, Accounts, Escalation, Building Manager, Other). Manual and CSV import.
Why
Reality is mixed — some contacts cover the whole customer (Accounts), others are site-specific (Site Manager). Roles are optional so you don't get blocked tagging everyone.
Build
  • New table: contacts — name, email, phone, role (nullable), customer_id, site_id (one of the two required)
  • "Add Contact" button on Customer detail and Site detail pages
  • Toolbar order on Contacts page: Import, Export, Add Contact
  • CSV import maps customer/site by name lookup (same pattern as Contract Scope import)
  • Show contacts on report cover page when relevant (configurable via Report Settings)

05Maintenance — edit after completion

Need to add work orders and notes to assets after the check is closed.

Maintenance
Any technician can amend, with full audit trail
Build
Decision
Completed checks can be re-opened by any user with write access. Every edit logged: user, timestamp, before-value, after-value. Status badge flips to "Amended" so it's visible on the report.
Why
Field reality — tech finishes a check, then a defect surfaces during pack-up or the customer adds a follow-up WO. Locking it would force ugly workarounds. Audit trail keeps you compliance-safe.
Build
  • Re-use the existing audit_log table — extend with action_type='amend'
  • "Re-open" button on completed check pages (write-access only)
  • Reports show "Amended on [date] by [user]" footer when applicable
  • Original PDF preserved as checks/{id}/v1.pdf, regenerated as v2.pdf after amend

06Maintenance attachments

You asked: where do they save and what for?

Maintenance
Categorized system: Evidence / Reference / Paperwork
Answer + Build
Today
Attachments save to the Supabase attachments bucket (private, RLS-protected by tenant). They're attached to whatever record you uploaded from but with no clear use.
Decision
Add a Type field. Each type has its own behaviour:
  • Evidence — photos/videos on tests & defects. Auto-included on PDF reports as evidence appendix.
  • Reference — SLDs, drawings, manuals. Pinned to the site detail page for techs to grab onsite. Not on reports.
  • Paperwork — POs, customer signoffs, dockets. Attached to work orders. Internal record only.
Build
  • Migration: add attachment_type enum + column to attachments
  • Backfill existing attachments based on parent record type (defect → Evidence, site → Reference, WO → Paperwork)
  • Upload modal forces a Type selection on every new upload
  • Site detail page gets a "Reference Documents" panel pulling all Reference attachments for that site

08Calendar

You said: catch what gets missed in Outlook. Today you use Outlook in SharePoint.

Calendar
A "miss-net" view, not an Outlook replacement
Build
Decision
Calendar page becomes a compliance-driven view of what should have happened but didn't:
  • Overdue PMs (job plan frequency exceeded, no completed check)
  • Upcoming PMs in next 30/60/90 days — auto-detected from job plans
  • Open work orders with target dates passed
  • Defects with no follow-up WO created after X days
Outlook keeps owning bookings & meetings. EQ Service catches the operational gaps Outlook can't see.
Why
Outlook is fine for "when am I doing the job". It's blind to "did I miss a job that's contractually owed". That's the gap — and only EQ Service has the data (job plans, contract scope, completed checks) to spot it.
Build
  • Rename Calendar → "Schedule & Gaps" (or similar — open to your call)
  • Three views: Overdue (red, top priority), This Week, Looking Ahead
  • Each row clickable through to the asset / customer / contract scope context
  • Optional: nightly job to push an Outlook .ics feed of upcoming PMs so they appear in your existing calendar too

09Testing reports — per check

"We need to be able to create reports per check."

Testing
One PDF per maintenance check — all assets, cover page, 3 styles
Build
Decision
A maintenance check (the container that wraps multiple ACB/NSX/General test records) produces a single PDF covering every asset inside it. Cover page, then a section per asset, all driven by Report Settings (summary / standard / detailed).
Why
Customers want one document per visit, not 20 separate PDFs. The check container is the natural unit — already groups the right assets, already has site & date.
Build

  • Per-asset section depth controlled by Report Settings complexity
  • Auto-store as checks/{id}/report.pdf, downloadable from the check & from the customer dashboard

10Defects

You asked: how does this get populated?

Defects
Auto from failed test items + manual add
Answer + Build
Today
Defects table exists but population is manual-only and barely used.
Decision
When a test item is marked "Not OK" or fails an electrical reading, a defect is auto-created with:
  • Linked check, asset, test record
  • Severity inferred from the test type (visual = low, electrical fail = high)
  • Defaults from the comment field on the failed item
A manual "Add Defect" button stays for things spotted off-test (e.g. damaged enclosure noticed during pack-up).
Build
  • Trigger function on test item insert/update — if status flips to fail, insert into defects
  • Reverse: if test item un-flips (e.g. amended after re-inspection), defect status → "Resolved"
  • Defects list shows source: "Auto from ACB test" vs "Manual"
  • Report includes a Defects appendix listing all auto+manual defects from the check

12Users — archive vs hard delete

You want to fully remove people from the list.

Users
Archive (any admin) + hard delete (super_admin only)
Build
Decision
Two-tier:
  • Archive — any admin or supervisor can do it. Hides from active list, blocks login, preserves all audit history with their name. "Show Archived" toggle to view.
  • Hard delete — super_admin only. Purges from auth.users; historical records replace user reference with a string ("Removed User"). Irreversible.
Why
Archive covers 95% of cases (someone leaves the team, no need to wipe their history). Hard delete is for genuine cleanup (test accounts, GDPR-style requests) and stays gated behind super_admin to prevent accidents.
Build
  • Add is_active to profiles (already exists pattern from job_plans/sites)
  • "Archive User" button on Users page — confirms, sets is_active=false, blocks login via auth hook
  • "Show Archived" toggle on Users page
  • "Delete Permanently" button visible only when role=super_admin — double-confirm modal
  • Audit log entries for both archive and delete actions
