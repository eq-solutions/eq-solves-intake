## Findings

### P1 — Customer portal sign-in entirely unreachable
**Surface 1.** `/portal/login` 307-redirects to `/auth/signin?next=%2Fportal%2Flogin`, and following the redirect lands on the **staff** sign-in page (h1: "Welcome to EQ Solves Service"), not the magic-link form. Same fate for `/api/portal/magic-link` (POSTs return 307 redirect, never reach the handler). Root cause: `lib/auth/mfa-routing.ts:PUBLIC_PATHS` doesn't list `/portal/login` or `/api/portal/magic-link`, so `proxy.ts:39` treats them as auth-gated. Net effect: zero customers can sign in to the portal in its current state.

This is auth-flow — queued, not fixed.

### P1 — Auth gate fix from PR #82 is not on `main`
**Surface 4.** The prompt asked me to verify behaviour assuming `proxy.ts` now reads `tenant_members.role`. It still reads `profiles.role` (lines 82-94). Commit `01d1388` exists locally but `git merge-base --is-ancestor 01d1388 origin/main` → NO. `git log origin/main -- proxy.ts` last touched the file on 2026-04-18 (`6d5de2f test(auth): add MFA AAL1-loop regression test`). The PR-#82-era admin-gate alignment never landed.

Implication: a user with `profiles.role = super_admin` but `tenant_members.role = supervisor` continues to slip past the proxy gate and only gets caught by the page-level `isAdmin()` check (the original Issue #19 behaviour). Auth-flow — queued.

### P1 — Notifications dispatcher firing but silently no-op
**Surface 2.** `cron.job_run_details` shows the `notifications-dispatcher` cron firing every 15 min and returning `succeeded`. But `dispatch_scheduled_notifications()` short-circuits at the `cron_secret IS NULL` guard (line: `IF v_secret IS NULL THEN ... RETURN; END IF;`). `vault.secrets` is empty (`SELECT name FROM vault.secrets;` → 0 rows). `app_url` IS set to `https://eq-solves-service.netlify.app`, just the secret is missing. The `notifications` table has 10 rows total, oldest 2026-04-08, newest 2026-04-27 — pre-dispatcher, all `email_sent = false`. Nothing has been dispatched since the cron started.

Documented in commit `31c7a27 docs(runbook): notifications cron bootstrap (CRON_SECRET + Vault)` — also not on `main`. So the operational runbook for the very bootstrap step that's blocking dispatch is itself stranded. Two-step fix: (a) generate hex secret + add to Netlify env + `vault.create_secret()`, (b) merge the runbook so future-Royce knows where to start.

### P2 — Defect notifications skip super_admin on `high` severity; no immediate email; no unsubscribe
**Surface 3.** Three sub-findings:
1. `lib/actions/defect-notifications.ts:27` routes `severity='critical'` to super_admin+admin+supervisor and **everything else** (low/medium/`high`) to admin+supervisor only. The `defects` table has 4 severities (`critical`, `high`, `medium`, `low`). A `high` defect doesn't escalate to super_admin — that may or may not be intentional.
2. The fanout only writes bell rows. There is no immediate email send for defects — the digest cron picks them up at the user's next slot (07:00 default). So a critical defect raised at 09:00 doesn't email anyone until ~22h later.
3. `lib/email/send-customer-monthly-summary.ts:135` ends with "Reply to this email to update your preferences." No functional unsubscribe link. The Australian Spam Act 2003 requires a one-click unsub for commercial electronic messages — once Phase C starts sending real customer mail at scale this is a compliance gap.
